## 行动实验提炼

### 实验 1：试用 Hermes Agent，对比是否更适合当前工作流
- **假设**：如果一个 Agent 更擅长自我总结和工作流优化，而不是依赖大量社区 Skill，那么它会比 OpenClaw 更贴合我当前的个人知识系统工作方式。
- **来源**：[[03@Diary/2026/04/2026-04-11]] :: 12:31
- **触发条件**：下周开始试用 Hermes Agent 时。
- **验证方式**：
  1. 用同一类任务分别在 Hermes 和 OpenClaw 里跑一遍。
  2. 记录是否需要反复调教、是否更容易形成稳定工作流、是否更符合“自己搭脚本串流程”的使用方式。
  3. 对比试用后的主观负担和产出稳定性。
- **预期价值**：判断当前主力 Agent 工具是否真的选对，避免把精力长期投在不贴合的工具栈上。

### 实验 2：把“工具适配性”作为选型标准固定下来
- **假设**：工具的优势不在于名气或功能数量，而在于是否贴合我“自己搭工作流、自己写脚本、少调教”的使用方式。
- **来源**：[[03@Diary/2026/04/2026-04-11]] :: 12:31
- **触发条件**：以后再遇到新工具或新 Agent 时。
- **验证方式**：
  1. 每次评估工具时固定问三个问题：是否少调教、是否适合脚本串联、是否能减少手工维护。
  2. 连续记录 3 次工具选型判断，看是否比“功能清单式比较”更有效。
- **预期价值**：减少被工具宣传带跑偏，提高选型命中率。

### 实验 3：为高频工作流预估 token 成本
- **假设**：工作流一旦铺开并每天执行，token 消耗会快速上升，必须提前做成本预估。
- **来源**：[[03@Diary/2026/04/2026-04-11]] :: 20:17
- **触发条件**：每当准备新增一个会频繁调用模型的工作流时。
- **验证方式**：
  1. 记录新增工作流前后的 token 消耗变化。
  2. 给每个工作流估一个日均/周均消耗。
  3. 跟实际消耗做对照，看看是否接近预估。
- **预期价值**：避免自动化铺开后才发现额度不够用，提前控制成本。

### 实验 4：把“稳定可靠的模型”列为工作流准入条件
- **假设**：免费模型如果不够稳定，最终会拖垮工作流；高质量模型虽然贵，但更适合长期依赖型流程。
- **来源**：[[03@Diary/2026/04/2026-04-11]] :: 20:17
- **触发条件**：工作流出现质量波动、输出不稳或频繁失败时。
- **验证方式**：
  1. 记录每个工作流对模型稳定性的要求。
  2. 如果某流程失败率高，比较换模型前后的效果。
  3. 观察是否“便宜模型能跑，但不能长期稳定跑”。
- **预期价值**：把模型质量问题前置，而不是在故障后被动补救。

### 实验 5：跟进 GitHub Copilot 作为 memory search embedding provider 的 PR
- **假设**：如果这个 PR 合并，Copilot 套餐就可能被纳入 memory search 的可用方案。
- **来源**：[[03@Diary/2026/04/2026-04-11]] :: 21:59
- **触发条件**：PR 状态发生变化，或新版本发布时。
- **验证方式**：
  1. 定期检查 PR 是否合并。
  2. 新版本发布后验证 memory search 是否可直接使用 Copilot provider。
  3. 记录实际可用性，不只看文档宣传。
- **预期价值**：判断现有套餐是否能覆盖更多内存相关能力，减少额外 provider 成本。

### 实验 6：试验把 memory wiki 纳入个人知识系统子模块
- **假设**：OpenClaw 内置 memory wiki 可以作为个人知识系统的一个子模块，而不只是独立功能。
- **来源**：[[03@Diary/2026/04/2026-04-11]] :: 22:51
- **触发条件**：开始研究 memory wiki 文档时。
- **验证方式**：
  1. 看它是否支持我想要的组织方式、索引方式和长期沉淀方式。
  2. 用一小组内容试着接入，观察是否比现有方案更顺手。
  3. 对比它和现有 LLMWiki / 工作流的边界。
- **预期价值**：判断是否能把一些知识管理能力下沉到工具内核，减少外部拼接成本。

### 实验 7：先学 OpenClaw 架构，再决定怎么用
- **假设**：理解 OpenClaw 的核心概念和架构，会直接提升我对它的使用质量，而不是只会表层操作。
- **来源**：[[03@Diary/2026/04/2026-04-11]] :: 22:53
- **触发条件**：准备继续深入使用 OpenClaw 时。
- **验证方式**：
  1. 按架构文档梳理一次它的核心组件、路由和知识能力。
  2. 用理解到的结构反推当前工作流哪里该放脚本、哪里该放 skill、哪里该放内置功能。
  3. 看理解架构后，是否减少误用和绕路。
- **预期价值**：把“会用工具”推进到“会判断工具边界”，减少盲用。

### 实验 8：观察潮白河边治理场景，形成对“管与放”边界的判断框架
- **假设**：现实中的空间治理，核心不是单纯严管或松管，而是边界控制是否能压住垃圾与失序。
- **来源**：[[03@Diary/2026/04/2026-04-11]] :: 19:11
- **触发条件**：以后再遇到类似公共空间、活动组织或秩序治理场景时。
- **验证方式**：
  1. 观察是否存在“太严不自由、太松出问题”的反复两难。
  2. 记录哪些约束方式真正减少了垃圾、破坏或失序。
  3. 对比不同治理策略的实际效果。
- **预期价值**：把一次生活观察转成可复用的现实判断框架。

### 实验 9：为多工作流系统做“成本—稳定性”双表评估
- **假设**：如果多个工作流都依赖模型，最关键的不是单个流程能不能跑，而是整套系统是否在成本和稳定性上可持续。
- **来源**：[[03@Diary/2026/04/2026-04-11]] :: 20:17、[[03@Diary/2026/04/2026-04-11]] :: 21:59
- **触发条件**：新增第二个及以上模型驱动工作流时。
- **验证方式**：
  1. 给每个工作流列出预计 token 消耗与失败风险。
  2. 运行一周后复盘实际成本和稳定性。
  3. 看是否需要减流、换模型或改成分级执行。
- **预期价值**：防止系统越自动化越失控，提前建立可持续边界。

### 实验 10：把“少调教”作为长期 Agent 评价指标
- **假设**：一个更好的 Agent，不是更会迎合，而是能更快总结出我常用的工作流，减少反复调教。
- **来源**：[[03@Diary/2026/04/2026-04-11]] :: 12:31
- **触发条件**：比较 Hermes、OpenClaw 或其他 Agent 时。
- **验证方式**：
  1. 记录每个工具从第一次使用到形成稳定流程所需的调教次数。
  2. 观察是否越用越顺，还是越用越需要人工纠偏。
  3. 把“调教成本”单独打分。
- **预期价值**：让工具选择更贴近真实使用体验，而不是只看技术参数。
