## 行动实验

### 1) 上下文串台排查实验
- **假设**：worklog summarize 变成周报，主要原因是多个工作流共用了 main agent，导致 prompt 上下文串台。
- **来源**：[diary] 08:55、10:10 关于工作日志总结内容跑偏及“上下文串台”的嫌疑分析。
- **触发条件**：worklog summarize 输出的结构、长度或主题明显偏离“当天工作日志总结”。
- **验证方式**：对比正常日期与异常日期的输入上下文、prompt 和 agent 路由，检查是否复用了同一上下文或同一主代理会话。
- **预期价值**：定位生成偏差的根因，避免工作日志和周报互相污染。

### 2) Main Agent 统筹路由实验
- **假设**：如果用一个 main agent 作为唯一入口，再由它分派 diary / coder / sage / writer 等子 agent，用户体验会更统一，同时能减少多入口负担。
- **来源**：[diary] 10:19 关于“超级助理”式 main agent 的场景设想。
- **触发条件**：当输入任务类型混杂，且希望用户只面对一个统一入口时。
- **验证方式**：先实现最小路由规则，再观察不同任务是否能被正确分派到对应子 agent，并顺利回传结果。
- **预期价值**：把多 Agent 变成“无感协作”，对用户来说只保留一个稳定接口。

### 3) 站点目录迁移实验
- **假设**：把 `09@LLMWiki-Site` 从 Obsidian 仓库迁移到独立 GitHub 目录后，主仓会更干净，展示层边界也更清晰。
- **来源**：[diary] 10:55 关于迁移 `09@LLMWiki-Site` 到 `F:\gitRoot\github\llmwiki-site` 的需求。
- **触发条件**：展示目录开始污染主仓，或需要独立管理发布产物时。
- **验证方式**：迁移后检查路径引用是否全部更新，确认主仓不再包含站点展示产物。
- **预期价值**：实现源内容与展示层分离，降低仓库噪音。

### 4) 分阶段自动提交实验
- **假设**：把“工作日志 → Wiki 同步 → Site 发布”拆成三个独立提交点，能更清楚地回溯每一步产物，也更利于排查问题。
- **来源**：[diary] 13:53 关于 workRoot / llmwiki-site 的分阶段自动化提交需求。
- **触发条件**：每天晚间定时任务执行时。
- **验证方式**：检查每天是否形成三次清晰提交，且每次提交只覆盖对应目录和对应阶段产物。
- **预期价值**：让系统演进过程可追踪、可回滚、可核查。

### 5) 空输入跳过实验
- **假设**：如果当天没有日记和工作日志，就让 summarize / sync 直接返回，能避免生成空壳内容。
- **来源**：[diary] 14:02 关于“没有输入就不生成”的脚本要求。
- **触发条件**：当天没有新增日记和工作日志条目时。
- **验证方式**：在空输入日测试脚本是否直接退出，不产生新的 wiki 内容或提交。
- **预期价值**：减少无价值产出，保持站点内容纯净。

### 6) 通用 git 提交脚本实验
- **假设**：一个通用的 `git-utils.js` 提交脚本，可以让不同项目的提交流程统一起来，减少重复脚本。
- **来源**：[diary] 15:47 关于实现通用 git 提交脚本的需求。
- **触发条件**：需要对任意项目执行标准化提交时。
- **验证方式**：用同一脚本对不同项目执行 commit，检查是否都能正确进入目标目录并完成提交。
- **预期价值**：统一提交入口，减少维护多套提交脚本的成本。

### 7) 只提交、不默认推送实验
- **假设**：把 push 从默认行为改成显式行为，能减少网络问题导致的自动化失败。
- **来源**：[diary] 15:55 关于 commit msg 乱码与 push 失败，要求暂时不再执行 push。
- **触发条件**：远程仓库网络不稳定，或用户希望先本地确认提交结果。
- **验证方式**：执行提交流程时默认只 commit，不 push；需要 push 时再显式开启。
- **预期价值**：让自动化更稳，避免网络问题拖垮整个提交流程。

### 8) 中文提交信息编码修复实验
- **假设**：如果脚本和 PowerShell 文件统一使用合适的 UTF-8 编码，中文 commit msg 就不会乱码。
- **来源**：[diary] 15:55 关于 commit msg 乱码问题的修复需求。
- **触发条件**：提交信息包含中文时出现乱码。
- **验证方式**：用中文 commit msg 执行完整提交流程，检查终端输出与 git 日志是否正常显示。
- **预期价值**：保证中文提交信息可读，避免自动化日志失真。
