# 2026-04-11 LLM 日报

## 今日一句话总结
今天把 OpenClaw 当作个人知识系统底座来重新审视了一遍：它的强项、成本、内存能力和未来扩展路径都被我重新校准了。

## 今日主题
- 个人工作流与 Agent 工具的匹配关系
- 模型调用成本与长期可持续性
- OpenClaw 的内存能力、Wiki 能力与架构理解

## 今日洞察
1. **工具强不强，不只看功能多不多，更看它是否适合我的使用方式。**  
   我现在主要是在自己搭工作流、自己写脚本、自己串知识管理流程，而不是大量依赖社区 Skill。这样一来，OpenClaw 最强的“社区 Skill 生态”优势并没有充分发挥。相对而言，Hermes Agent 这种更偏“自动总结工作流、持续自我优化”的方向，反而更贴近当前需求。

2. **工作流铺开后，token 成本会从“可忽略”变成“结构性问题”。**  
   以前 GitHub Copilot 额度主要够写代码用，现在接入 OpenClaw 后，额度开始明显紧张。这说明一旦多个流程每天都跑，模型消耗会非常可观，而且高质量、稳定的模型往往又更贵。成本问题不是边角料，而是后续系统设计必须正面面对的约束。

3. **很多能力不是“没有”，而是“还没接上”。**  
   OpenClaw 的 MEMORY SEARCH 目前还需要单独的 Agent Provider，暂时不能直接用 GitHub Copilot 套餐；但已经看到有 PR 在推进支持 Copilot 作为 embedding provider。这种信息很关键：它说明当前限制可能是版本或集成层面的，不是能力本身不存在。

4. **内置 memory wiki 可能比我想象中更有用。**  
   今天才意识到 OpenClaw 自带 memory wiki，这意味着它不只是任务/对话工具，也可能成为个人知识系统的子模块。这个方向值得认真研究，而不是只把它当作聊天壳子。

5. **理解架构本身，会直接影响工具使用质量。**  
   我今天明确觉得，学习 OpenClaw 的核心概念和架构不是“可选项”，而是提高使用效率的前提。知道它怎么分层、怎么路由、怎么组织记忆，才能判断哪些地方该靠脚本，哪些地方该靠 skill，哪些地方该靠内置能力。

## 今日建议
- **先试用，再判断。** Hermes Agent 先实际试一轮，不要只凭感觉下结论。
- **把 token 成本纳入工作流设计。** 以后每加一个自动化流程，都要先问：它每天会吃掉多少模型额度？
- **持续跟进 OpenClaw 的 PR 和版本更新。** 尤其是 memory 搜索和 embedding provider 相关支持。
- **认真研究 memory wiki 和 architecture 文档。** 这两块可能决定后续个人知识系统怎么搭。
- **不要盲目重构成 skill。** 在当前阶段，脚本底座仍然更稳，skill 更适合做包装和迁移。
