- 课题研究系统的工作流设计
- 课题、素材、草稿、定稿的分层管理
- wiki 系统跑稳后的下一步扩展

## 今日洞察
1. **工作流再好，也离不开模型的能力和稳定性。**  
   你已经明确意识到，工作流的质量上限取决于模型本身：模型能力不够，流程质量会打折；模型不稳定，工作流就没有稳定价值。这说明你已经不再只看流程设计，而是在看“流程 + 底座模型”的整体可靠性。

2. **核心工作流不能绑定死一个 Agent / Model。**  
   你担心一旦主 Agent 或主模型出问题，自己会非常被动，所以明确提出核心工作流必须有 Plan B，备用 Agent 要能接替。这种思路很像岗位备份机制：关键流程不能只有单点依赖。

3. **你对系统设计的关注点已经从“能不能跑”转向“能不能长期管理”。**  
   工作日志、任务、日记、wiki 这套系统已经跑起来了，你现在不急着重造，而是先跑一阵观察。说明你已经开始接受“先运行，再看问题，再迭代”的节奏。

4. **课题研究系统需要围绕“一个课题一个文件”来组织。**  
   你已经想清楚，课题不能散落在很多文件里，否则难以管理。一个课题文件里要有名称、边界定义、日常思考条目和总提示词，这样课题的思路才会收束，不至于碎片化。

5. **课题研究系统本质上是“人工消化 + LLM 辅助”的闭环。**  
   你不是想让 LLM 直接替你完成研究，而是让它先帮你整理草稿，再由你审阅修改、形成定稿。也就是说，LLM 是加速器，不是替代者；研究的主导权仍然在你手里。

6. **你对知识系统的目标越来越明确：让它服务“课题推进”，而不是只服务记录。**  
   你已经把下一步重心放到课题研究系统上，说明你希望知识系统不止是存储和回看，而是能真正推动具体研究产出。

## 今日建议
- 给核心工作流加一个明确的备用 Agent / Model 方案，别让主流程单点失效。
- 课题研究系统优先搭一个最小可用版本：一个课题文件 + 一个素材库 + 一个草稿库。
- 每个课题尽量收敛到单文件管理，避免思路分散在多个地方。
- 素材收集和人工消化分开记录，别让原始素材和你的思考混在一起。
- LLM 只负责草稿和结构化辅助，最终结论仍由你审阅定稿。
- 先让现有工作流稳定跑一阵，再把精力集中到课题研究系统上，减少同时开太多新坑。
