# 08@LLMWiki 长期架构说明

## 目标
让 LLMWiki 变成一个低心智负担、可持续积累的个人知识系统：

- 平时只管记日记
- 系统自动整理
- 打开索引页即可查看
- 画像和主题页不是当天快照，而是长期累积结果

---

## 分层设计

### 1. 原始层
`03@Diary`、`07@WorkLog`

- `03@Diary`：保存原始成功日记
- `07@WorkLog`：保存每日工作日志
- 两者共同作为事实来源
- 不承担知识总结职责

### 2. 派生层（日更）
`08@LLMWiki/daily`、`08@LLMWiki/ideas`

- `daily`：当天摘要、洞察、建议
- `ideas`：当天提炼出的行动实验
- 可按天重建
- 不要求长期继承

### 3. 长期知识层
`08@LLMWiki/profile`、`08@LLMWiki/themes`

- `profile`：长期画像
- `themes`：长期主题档案
- 必须继承历史内容
- 不能只看当天日记

### 4. 导航层
`08@LLMWiki/index`

- 只做入口
- 不承载核心知识
- 负责快速查看最新内容

---

## 画像页原则

`profile/个人画像.md` 是长期累计文档，不应该被当天内容覆盖。

### 更新方式
每次同步时：
1. 读取旧画像
2. 读取当天新增成功日记
3. 让模型做增量更新
4. 输出更新后的画像
5. 另存历史快照

### 画像应包含
- 画像结论
- 稳定特征
- 最近变化
- 证据
- 置信度
- 仍待验证的问题
- 最近更新时间

---

## 主题页原则

`themes/*.md` 是主题专题档案，不是当天分类结果。

### 更新方式
每次同步时：
1. 读取旧主题页
2. 读取当天相关条目
3. 让模型把新增证据合并进主题页
4. 输出更新后的专题页
5. 另存历史快照

### 主题页应包含
- 主题定义
- 核心结论
- 高频现象
- 最近变化
- 证据
- 可尝试策略
- 尚未确认的问题
- 最近更新时间

---

## 历史快照

为了可追溯，长期知识页建议额外保留快照：

- `profile/history/YYYY-MM-DD.md`
- `themes/history/<主题>/YYYY-MM-DD.md`

用途：
- 回看画像怎么变
- 回看主题如何演进
- 排查某次判断为什么变了

---

## 自动化运行方式

### 每天
```bash
node scripts/llmwiki.js sync --date YYYY-MM-DD
```

建议行为：
- 生成日报
- 更新主题页
- 更新画像页
- 生成想法页
- 更新索引页
- 写入历史快照

### 每周
```bash
node scripts/llmwiki.js weekly --date YYYY-MM-DD
```

### 每月
```bash
node scripts/llmwiki.js monthly --date YYYY-MM-DD
```

---

## 使用体验目标

你只需要：

1. 记成功日记
2. 想看时打开索引页
3. 需要时回看主题或画像

不要手工维护主题页和画像页；它们应该自己长出来。

---

## 设计底线

- 日更内容可以覆盖
- 长期内容不能只看当天
- 旧判断不能无声消失
- 所有长期结论都要保留证据链
