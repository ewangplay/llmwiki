## 重复模式
- **先发现问题，再把问题上升为架构问题**
  - 4/10 工作日志总结串到周报，不只被当作单次 bug，而是被判断为 main agent 上下文串台。
  - 4/15 画像频繁更新，不只是“更新太多”，而是被重新定义为“稳定层不应日更”。

- **持续把流程拆分成可管理的层级**
  - 日报、周报、月报、画像逐层分离。
  - 工作日志、成功日记、主题页、展示站点各自分工。

- **偏好低心智负担的自动化**
  - 你希望自己只提供输入，其余分类、总结、提交、分流交给系统。
  - 但前提是系统必须稳定、可回溯、不会产出空壳内容。

- **对“无意义输出”保持强排斥**
  - 空输入不生成。
  - 模型失败不 fallback 生成。
  - 这说明你更在意结果真实有效，而不是流程表面完整。

- **工具判断标准高度一致**
  - 你反复用同一套标准看工具：是否贴合工作流、是否稳定、是否少调教、是否可持续、是否值得成本。
  - Hermes 之所以被看好，不是因为“新”，而是因为更贴合你现在的脚本化工作方式。
